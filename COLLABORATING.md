# Collaborating on Dockable

Thanks for your interest in contributing! Dockable is a macOS-style Dock for Windows 11 —
a WPF app that leans heavily on Win32 interop, so contributions range from pure C#/XAML work
to deep shell/DWM plumbing. This guide explains how to contribute in a way that fits the
project's conventions and gets your work merged smoothly.

## Ways to contribute

- **Bug reports** — open an issue with your Windows version/build, what you did, what you
  expected, and what happened. If the app crashed, attach `%APPDATA%\Dockable\crash.log`.
  A short screen recording is worth a thousand words for animation/layout glitches.
- **Feature work** — check the open issues and the roadmap below first, then open an issue
  to discuss the approach *before* writing code. Dockable is opinionated about matching
  macOS Dock behavior, so agreeing on the UX up front saves everyone time.
- **Localization** — the app ships in English, Português (Brasil), Español, Українська, and
  中文 (Simplified). Fixing awkward translations or adding a new language is very welcome
  (see [Localization](#localization) below).
- **Docs** — the README and in-code docs can always be clearer.

### Good areas to pick up

Known open TODOs, roughly in order of impact:

- **Position on screen: Left / Right edges** — the setting exists but only Bottom is implemented.
- **Exact secondary-monitor placement** (currently approximate).
- **Auto-remove stale minimized tiles** when an app exits while minimized
  (needs an `EVENT_OBJECT_DESTROY` hook).
- **UWP/Store pin matching** — pins without an exe target may not match their running window.
- Suppress the running dot for an app whose only window is minimized into a tile.

## Prerequisites & setup

- **Windows 11 x64** (the app targets `net9.0-windows10.0.22621.0`; much of the interop is
  Windows-11-specific — taskbar internals, DWM attributes, XAML-island tray, etc.).
- **.NET 9 SDK**.
- Any editor works; Visual Studio 2022 or Rider give the best WPF/XAML experience.

```sh
git clone https://github.com/jezzlucena/Dockable.git
cd Dockable
dotnet build src/Dockable/Dockable.csproj -c Debug
dotnet run   --project src/Dockable/Dockable.csproj
```

Things to know when developing:

- **Only one dock runs** (single-instance mutex). A second launch bows out silently.
- **Stop the running `Dockable.exe` before rebuilding** — it locks the output, and the build
  fails at the *copy* step even though compilation succeeded. Look for `error CS` lines to
  tell a real compile error from a file-lock failure.
- Killing the dock is safe: an out-of-process watchdog restores the Windows taskbar's
  pre-launch state even after a hard kill.
- **Settings live at `%APPDATA%\Dockable\settings.json`.** If your change seeds or migrates
  settings, test with both a fresh file and your existing one — and never ship code that
  silently discards a user's settings.

## Project orientation

The README's [Project layout](README.md#project-layout) section maps the tree. The short version:

- `DockWindow` is the dock itself (positioning, magnification loop, drag, menus, minimize
  orchestration); `MenuBarWindow` is the optional top menu bar; `SettingsWindow` is Dock
  Preferences.
- `ViewModels/DockLayoutEngine` owns the fisheye magnification and drag layout math.
- `Interop/` is all the Win32 plumbing; `Genie/` is window capture + the minimize animators;
  `Shell/` is icon extraction and folder listings; `Localization/` is the string tables.

`CLAUDE.md` at the repo root is a deep architectural log — decisions, gotchas, and *why*
things are built the way they are. It's dense, but if you're touching magnification,
minimize/restore, AppBar reservation, or the taskbar mirror, read the relevant section first;
most of the non-obvious constraints are documented there.

## Coding conventions

- **C# 12**, nullable enabled, file-scoped namespaces, implicit usings. Match the style of
  the file you're in.
- **MVVM-ish**: `CommunityToolkit.Mvvm` (`[ObservableProperty]` etc.) for view-models;
  view-specific orchestration lives in the window code-behind by design.
- Short XML docs on public members. Inline comments only where intent is non-obvious —
  especially *why* a Win32 call behaves the way it does.
- **Keep per-frame work off WPF layout passes and allocation-free.** The magnification render
  loop runs every frame while hovered; don't introduce per-frame `new`, per-frame DPI queries,
  or per-frame COM calls. The 1-second taskbar refresh is similarly hot — anything newly
  per-window-per-tick needs caching (see `TaskbarApps.IdentityCache` for the pattern).
- Shell icon extraction runs **off the UI thread** and every produced `BitmapSource` must be
  `Freeze()`d.

### Win32 interop (CsWin32)

P/Invoke is source-generated by **CsWin32** from `src/Dockable/NativeMethods.txt` (one API or
type per line). To add a Win32 call:

1. Add its name to `NativeMethods.txt`.
2. Build, then call it as `Windows.Win32.PInvoke.<Name>`.
3. Signatures vary (`in`/`out`/pointers/`Span` overloads) — if unsure, build once with
   `-p:EmitCompilerGeneratedFiles=true` and read the generated code under
   `obj/.../generated/Microsoft.Windows.CsWin32/`.

Prefer CsWin32 over hand-written `DllImport`; if an API isn't in the Win32 metadata at all,
a manual declaration is fine — note why in a comment. CsWin32 types are `internal`, so
anything exposing them in a signature must be internal too.

### Localization

Every user-facing string must be localized:

- Strings live in `Localization/LocData.cs` — per-language tables keyed by string keys.
- **Add every new key to *all* language tables** (English is the fallback, but don't leave
  holes on purpose). Machine-translated entries are acceptable for languages you don't speak;
  flag them in the PR so a native speaker can review.
- XAML uses the `{loc:Loc Key=…}` markup extension; code-built UI uses `Loc.T("Key")`.
  Language changes apply **live** — menus built once must rebuild on `Loc.LanguageChanged`.
- Brand/tech names ("Dockable", C#, WPF, CsWin32, …) stay untranslated.

### Accessibility

Interactive elements need UI Automation support. Dock items, flyout rows, and code-built
click targets already expose localized names and Invoke via the peers in `Accessibility/` —
extend those patterns (`DockItemElement`, `InvokableRow`/`InvokableCell`) rather than adding
unnamed clickable surfaces.

## Testing your changes

There is no automated UI test suite — the dock's behavior (magnification feel, animation
timing, AppBar reservation, DPI handling) is verified by hand. Before opening a PR:

1. **`dotnet build` must succeed with 0 errors and no new warnings.**
2. **Exercise your change in the running app**, plus the neighbouring behavior it could
   plausibly break. High-risk areas and their usual regressions:
   - *Layout/magnification*: hover jitter, window width stepping, the idle click-through clip
     (`SetWindowRgn`) cutting off magnified icons.
   - *Minimize/restore*: rapid consecutive minimizes, restore-from-taskbar/Alt+Tab sync,
     windows left stranded after quitting the dock.
   - *Taskbar mirror*: pin/unpin/reorder, running-dot state, the 1 s refresh not churning the UI.
   - *Anything themed*: check Light, Dark, and Auto.
   - *Anything with text*: check at least one non-English language.
3. **Test at more than one DPI** if your change touches positioning or capture (the app is
   per-monitor-v2 DPI aware).
4. For visual/animation changes, include a short **screen recording or before/after
   screenshots** in the PR — reviewers can't feel your animation from a diff.

## Pull request guidelines

- **One logical change per PR.** Small, focused PRs get reviewed fast; grab-bag PRs stall.
- Discuss non-trivial features in an issue first.
- Write a clear description: what changed, why, how you tested it (Windows build, DPI,
  themes/languages exercised), and any known limitations.
- Don't reformat code you aren't otherwise touching, don't bump package versions as a side
  effect, and don't commit changes to `settings.json` seeds without calling them out.
- Keep the docs honest: if your change alters behavior described in the README or adds a
  setting, update the README (and the settings table in `CLAUDE.md`) in the same PR.
- New assets (icons, sounds) must be **embedded resources** — the portable single-file build
  guarantees nothing sits beside the exe.

### Commit messages

Short imperative subject line ("Add Left-edge dock positioning"), with a body explaining the
*why* when it isn't obvious. Reference issues with `#123`.

## Things we deliberately won't change

A few decisions are settled — PRs that reverse them won't be merged without a very strong case:

- **Pins are dock-owned and never written back to the Windows taskbar.** Windows blocks
  programmatic taskbar pinning/reordering; the dock keeps its own ordered pin list instead.
- **Window capture uses `BitBlt(CAPTUREBLT)`**, not `PrintWindow` (which returns black for
  composited apps).
- **The window's size is mirrored from the view-model in code-behind**, not bound in XAML
  (WPF's `Window.Width/Height` bindings are unreliable) — don't reintroduce the bindings.
- **No live system-tray icon replication in the menu bar** — on current Windows 11 the classic
  notification-area windows are gone and the XAML-island tray exposes nothing to UIA; the
  Quick Settings / Notifications shortcuts are the supported substitute.

## Questions & conduct

Be kind, be constructive, assume good faith. Disagreements about UX get settled by "what does
the macOS Dock do?" more often than not. For anything unclear, open an issue — a five-minute
question beats a week of building the wrong thing.
